# Payload + Next.js Website Conventions

Opinionated rules for building a content-driven website on **Payload v4** running natively inside **Next.js (App Router)**. These rules encode the conventions of the official `templates/website` starter. Follow them unless the user explicitly overrides.

When the work matches a heavier workflow, use the companion skill instead of improvising:

- **scaffold-payload-collection** ‚Äî create a Collection or Global (access, slug, SEO, drafts, preview, revalidation).
- **add-payload-block** ‚Äî add a layout-builder Block (`config.ts` + `Component.tsx` + register + render).
- **payload-frontend-fetch** ‚Äî fetch Payload data in Server Components with correct draft/preview/cache handling.
- **payload-migrate-deploy** ‚Äî DB migrations workflow and the production deploy checklist.

## The Stack (do not deviate without asking)

| Concern        | Choice                                                                                  |
| -------------- | --------------------------------------------------------------------------------------- |
| Runtime        | Node `>=24.15.0`, **pnpm**                                                               |
| Framework      | Next.js 16 App Router, React 19                                                          |
| CMS            | `payload` v4 (all `@payloadcms/*` packages pinned to the **same** version)              |
| Database       | **Postgres** via `@payloadcms/db-postgres` (Drizzle under the hood)                      |
| Rich text      | `@payloadcms/richtext-lexical`                                                           |
| File storage   | `@payloadcms/storage-vercel-blob` (local disk in dev when token absent)                 |
| Styling        | Tailwind CSS v4 + shadcn-style primitives (`@radix-ui/*`, `cva`, `clsx`, `tailwind-merge`, `lucide-react`, `geist`) |
| Plugins        | `plugin-seo`, `plugin-redirects`, `plugin-search`, `plugin-nested-docs`, `plugin-form-builder` |
| Live preview   | `@payloadcms/live-preview-react`                                                         |
| Types          | TypeScript 6, strict                                                                     |

**Version lockstep:** every `@payloadcms/*` package and `payload` itself must share the exact same version. Upgrade them together, never individually.

## Project Structure

The Payload config lives in `src/` and the app is split into two Next.js route groups: `(payload)` (admin + API, generated, leave alone) and `(frontend)` (your site).

```
src/
‚îú‚îÄ‚îÄ payload.config.ts            # central config ‚Äî db, collections, globals, plugins, storage, editor
‚îú‚îÄ‚îÄ payload-types.ts             # GENERATED ‚Äî never edit by hand; run `pnpm generate:types`
‚îú‚îÄ‚îÄ app/
‚îÇ   ‚îú‚îÄ‚îÄ (payload)/               # Admin Panel + REST/GraphQL. Generated by Payload ‚Äî do not edit.
‚îÇ   ‚îî‚îÄ‚îÄ (frontend)/              # Your website (RSC). Pages, layouts, preview routes, sitemaps.
‚îÇ       ‚îî‚îÄ‚îÄ next/preview/route.ts
‚îú‚îÄ‚îÄ collections/<Name>/index.ts  # one folder per collection
‚îú‚îÄ‚îÄ globals/<Name>/index.ts      # Header, Footer, site settings
‚îú‚îÄ‚îÄ blocks/<Name>/               # layout-builder blocks: config.ts + Component.tsx
‚îÇ   ‚îî‚îÄ‚îÄ RenderBlocks.tsx         # maps blockType -> Component
‚îú‚îÄ‚îÄ heros/                       # hero variants + RenderHero.tsx
‚îú‚îÄ‚îÄ access/                      # anyone.ts, authenticated.ts, authenticatedOrPublished.ts
‚îú‚îÄ‚îÄ fields/                      # reusable field factories (slugField, link, linkGroup, defaultLexical)
‚îú‚îÄ‚îÄ hooks/                       # shared collection/field hooks (revalidate, populatePublishedAt)
‚îú‚îÄ‚îÄ components/                  # frontend React components (RichText, Media, Link, ...)
‚îú‚îÄ‚îÄ utilities/                   # generateMeta, generatePreviewPath, getMediaUrl, getURL, ...
‚îú‚îÄ‚îÄ plugins/index.ts             # all plugins configured in one place
‚îî‚îÄ‚îÄ migrations/                  # generated SQL migrations (committed)
```

**Component file convention:** one folder per component ‚Äî `ComponentName/index.tsx` (+ `index.scss` if needed). Re-export through barrel `index.ts` files when grouping. Do **not** drop `Foo.tsx`, `Bar.tsx` into a shared folder.

## Payload Config Rules

- The config is server-only. It must never import client code (`.scss`, React client components, browser APIs). The admin layer derives a stripped client config automatically.
- Keep `payload.config.ts` thin: import collections/globals/blocks/plugins from their own files. Never inline large collection definitions.
- `secret` comes from `process.env.PAYLOAD_SECRET` ‚Äî long, random, never committed.
- Database:
  ```ts
  db: postgresAdapter({ pool: { connectionString: process.env.DATABASE_URI } })
  ```
  Leave Drizzle `push` enabled in development (default). See **payload-migrate-deploy** before touching production.
- Always pass `sharp` so image resizing / focal points work.
- Set `admin.user` to your auth-enabled collection (usually `users`).
- Register storage under the top-level `storage` key (not `plugins`):
  ```ts
  storage: [ vercelBlobStorage({ enabled: Boolean(process.env.BLOB_READ_WRITE_TOKEN),
    collections: { media: true }, token: process.env.BLOB_READ_WRITE_TOKEN }) ]
  ```
  Gate it on the env var so local dev falls back to disk storage automatically.

## TypeScript & Generated Types

- `src/payload-types.ts` is generated. **Never hand-edit it.** Run `pnpm generate:types` after any schema change (new collection, field, block, global).
- Import generated types everywhere you touch data:
  ```ts
  import type { Page, Post, CallToActionBlock } from '@/payload-types'
  ```
- Local API calls are auto-typed from these ‚Äî `payload.find({ collection: 'pages' })` returns `Page`. Don't re-declare shapes by hand.
- Blocks always generate a top-level interface from their slug; set `interfaceName` on each block to get a stable, importable name (e.g. `interfaceName: 'CallToActionBlock'`).
- The `generate:types` npm script must point Payload at the config:
  ```json
  "generate:types": "cross-env PAYLOAD_CONFIG_PATH=src/payload.config.ts payload generate:types"
  ```

## Coding Patterns (non-negotiable)

- **Object params:** `fn({ name }: { name: string })`, never `fn(name)`. Improves forwards/backwards compatibility.
- Prefer **types over interfaces** (except when extending an external interface).
- Prefer **pure functions** over classes (classes only for errors/adapters). Return mutated objects rather than mutating in place.
- Export-first ordering: public exports at the top of the file, helpers below.
- `import type` for types, regular `import` for values ‚Äî separate statements.
- Boolean names are prefixed: `isPublished`, `hasMedia`, `canEdit`, `shouldRefresh`.
- Prefer self-describing names over a generic name plus an explanatory comment.
- **Labels/translations:** render config labels with `getTranslation` from `@payloadcms/translations` ‚Äî never hand-roll string/function/object branching.
- **Never pass inline array/object literals to React hooks** ‚Äî memoize for a stable reference:
  ```ts
  // BAD: new array every render
  useThing({ filter: [slug] })
  // GOOD
  const filter = useMemo(() => [slug], [slug])
  useThing({ filter })
  ```
- Logger: `payload.logger.error({ msg: '...', err })` ‚Äî use `msg`/`err` keys; never pass the error as a second positional arg.

## Access Control (security-critical)

Centralize access functions in `src/access/` and reuse them. The canonical trio:

```ts
export const anyone: Access = () => true
export const authenticated: Access = ({ req: { user } }) => Boolean(user)
export const authenticatedOrPublished: Access = ({ req: { user } }) => {
  if (user) return true
  return { _status: { equals: 'published' } } // query constraint, not a boolean
}
```

Rules:

- **Public-readable content collections** (pages, posts) use `read: authenticatedOrPublished`, and `create`/`update`/`delete: authenticated`.
- Returning a **query constraint** (a `where` object) from `read`/`update` filters rows instead of all-or-nothing ‚Äî this is how you hide drafts from the public and how you scope "users see only their own records."
- Access functions can run with `id`/`data`/`doc` **undefined** (the Access Operation that builds the admin UI). Guard those before using them.
- Drafts are NOT hidden automatically. The `draft: true` read param does not enforce `_status` ‚Äî you must gate it in `read` access. This is the most common Payload security bug.
- **Local API skips access control by default.** Any code path reachable by an end user (Server Functions, route handlers, custom endpoints) MUST pass `overrideAccess: false` and the authenticated `user`:
  ```ts
  // INSECURE ‚Äî bypasses all access control
  await payload.find({ collection: 'posts' })
  // SECURE
  await payload.find({ collection: 'posts', overrideAccess: false, user })
  ```
  The only place `overrideAccess: true` is acceptable is trusted server-internal work (seeding, migrations, system jobs).

## Server Functions & Endpoints

- Mutations triggered from the client go through Next.js **Server Functions** (`'use server'`), never an exposed REST/GraphQL surface.
- Get Payload with `const payload = await getPayload({ config })` (import `config from '@payload-config'`).
- Wrap Local API calls in try/catch, log server-side, and return a structured `{ error }` instead of leaking raw errors. Never return sensitive fields.
- For auth flows use the ready-made `login` / `logout` / `refresh` from `@payloadcms/next/auth` wrapped in a thin `'use server'` helper (they handle cookies/sessions).
- Build admin/API URLs with `formatAdminURL`; build query strings with `qs-esm` ‚Äî never concatenate `where[...]=` by hand.

## Drafts, Preview & Revalidation

Content collections enable versions + drafts and wire up live preview:

```ts
versions: {
  maxPerDoc: 50,
  drafts: { autosave: { interval: 100 }, schedulePublish: true },
}
```

- `autosave.interval: 100` keeps **live preview** responsive (server-side live preview refreshes on save).
- `_status` is injected automatically (`draft` | `published`); never define it yourself.
- **Draft preview** flows through `admin.preview` ‚Üí `/next/preview` route (verifies `PREVIEW_SECRET`, authenticates the user, calls `draftMode().enable()`) ‚Üí the page reads `draftMode()` and queries with `draft: isEnabled`.
- After mutations, revalidate the affected paths/tags with `afterChange`/`afterDelete` hooks (`revalidatePath`/`revalidateTag`). Keep these hooks in the collection's `hooks/` folder.
- `schedulePublish` requires the **Jobs Queue** to be running, or scheduled events never fire.

## Frontend Data Fetching (RSC)

- Fetch with the **Local API** directly in Server Components ‚Äî no HTTP round-trip. This is Payload's biggest perf win; do not stand up REST/GraphQL fetches for your own pages.
- Wrap query helpers in React `cache()` so `generateMetadata` and the page component share one DB hit.
- For list/detail pages mirror the template: `generateStaticParams` (with `select` to fetch only slugs, `overrideAccess: false`), a `cache()`-wrapped `queryBySlug`, `draftMode()` to switch draft/published, `generateMeta` for metadata.
- **Depth:** rich text containing uploads/links needs a high enough `depth` to fully populate, or the JSX converter breaks. Use `defaultPopulate` on referenced collections to control what gets pulled when relationships expand.
- Render Lexical with `<RichText>` from `@payloadcms/richtext-lexical/react`, supplying `internalDocToHref` (internal links) and block converters keyed by slug.
- See **payload-frontend-fetch** for the full pattern.

## Database Migrations (Postgres)

- **Dev:** Drizzle `push` keeps your local DB in sync automatically. Treat local DB as a sandbox. Do **not** mix `push` and migrations against the same local DB.
- **Before shipping a schema change:** `pnpm payload migrate:create <name>`, review the generated SQL, commit it.
- **Prod:** run `payload migrate` in CI before `next build`. Standard build script: `"ci": "payload migrate && pnpm build"`.
- Never edit an already-applied migration; create a new one.
- See **payload-migrate-deploy** for the end-to-end workflow and deploy checklist.

## Verifying Changes

- Dev server: `pnpm dev`. Admin at `/admin`, site at `/`.
- After any config/field change: `pnpm generate:types`, then `pnpm lint`.
- Production-bundling issues (client code evaluated on the server) only surface in a prod build ‚Äî test real builds, not just `pnpm dev`, before declaring a bundling fix done.
- Don't claim a feature works without exercising it (admin save + frontend render, including draft preview where relevant).
